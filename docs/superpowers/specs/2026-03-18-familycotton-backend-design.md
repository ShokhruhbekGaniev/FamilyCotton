# FamilyCotton Backend — Design Spec

**Date:** 2026-03-18
**Stack:** Go 1.22+ | chi v5 | PostgreSQL 16 (pgx v5) | goose | JWT | Docker Compose
**Architecture:** Classic Layered (handler → service → repository)

---

## 1. Decisions Log

| Question | Decision |
|----------|----------|
| Implementation approach | 5 phases, sequential |
| total_debt fields | Denormalized, updated on each transaction |
| Employee permissions | Shifts, sales, returns, products (read+create), clients |
| Owner permissions | Full access |
| photo_url | Plain string, no upload |
| JWT tokens | Access: 1h, Refresh: 30 days |
| Pagination | Offset-based: `?page=1&limit=20` |
| Inventory check completion | Auto-correct stock quantities |
| Exchange rate | User inputs manually per transaction |
| Architecture | Classic Layered (handler → service → repository) |

## 3. Implementation Phases

### Phase 1 — Foundation

- Project scaffolding, go.mod, directory structure
- Configuration loading from env
- Docker Compose: Go app (port 8082) + PostgreSQL 16 (port 5434)
- Dockerfile: multi-stage build (golang:1.22-alpine → alpine)
- Goose migrations for all 17 tables
- JWT authentication (login, refresh, me)
- Users CRUD (owner only)
- Middleware: auth (JWT validation), CORS, request logging (slog), RBAC (RequireRole)
- Chi router with base route structure
- Unified error handling and response format

### Phase 2 — Catalogs

- Products CRUD with filters (?search, ?supplier_id, ?brand)
- Suppliers CRUD
- Clients CRUD
- Creditors CRUD
- Offset-based pagination on all list endpoints

### Phase 3 — Sales Operations

- Shifts: open/close with full business logic (aggregation, safe transactions, owner debts)
- Sales: split payments, qty_shop deduction, client.total_debt update
- Sale returns: full / exchange / exchange_diff with stock and safe adjustments
- Client payments: debt repayment

### Phase 4 — Procurement & Stock

- Purchase orders with items (stock arrival to shop/warehouse)
- Supplier payments (money / product return)
- Creditor transactions (receive/repay with exchange rate)
- Stock transfers (warehouse ↔ shop)
- Inventory checks with auto-correction on completion

### Phase 5 — Safe & Dashboard

- Safe transactions log (all monetary movements)
- Safe balance endpoint (cash / terminal / online)
- Owner debts + owner deposit
- Dashboard: revenue, profit, stock-value, sales-by-supplier, paid-vs-debt

## 4. Database

17 tables, all using:
- UUID primary keys (generated by PostgreSQL via `gen_random_uuid()`)
- TIMESTAMPTZ for dates (with `DEFAULT NOW()`)
- DECIMAL(15,2) for monetary amounts

### Tables

1. **users** — id, name, login (unique), password_hash (bcrypt), role (owner|employee), created_at
2. **suppliers** — id, name, phone, notes, total_debt, created_at
3. **creditors** — id, name, phone, notes, total_debt, created_at
4. **clients** — id, name, phone, total_debt, created_at
5. **products** — id, sku (unique), name, brand, supplier_id (FK), photo_url, cost_price, sell_price, qty_shop, qty_warehouse, created_at
6. **shifts** — id, opened_by (FK), closed_by (FK, nullable), opened_at, closed_at (nullable), total_cash, total_terminal, total_online, total_debt_sales, status (open|closed)
7. **sales** — id, shift_id (FK), client_id (FK, nullable), product_id (FK), quantity, unit_price, total_amount, paid_cash, paid_terminal, paid_online, paid_debt, created_at
8. **sale_returns** — id, sale_id (FK), product_id (FK), new_product_id (FK, nullable), quantity, return_type (full|exchange|exchange_diff), refund_amount, surcharge_amount, created_at
9. **purchase_orders** — id, supplier_id (FK), payment_type (cash|debt), total_amount, paid_amount, status (paid|partial|unpaid), created_at
10. **purchase_order_items** — id, purchase_order_id (FK), product_id (FK), quantity, unit_cost, destination (shop|warehouse)
11. **supplier_payments** — id, supplier_id (FK), purchase_order_id (FK, nullable), payment_type (money|product_return), amount, returned_product_id (FK, nullable), returned_qty (nullable), created_at
12. **creditor_transactions** — id, creditor_id (FK), type (receive|repay), currency (UZS|USD), amount, exchange_rate, amount_uzs, created_at
13. **client_payments** — id, client_id (FK), amount, payment_method (cash|terminal|online), created_at
14. **stock_transfers** — id, product_id (FK), direction (warehouse_to_shop|shop_to_warehouse), quantity, created_at
15. **safe_transactions** — id, type (income|expense|transfer), source, balance_type (cash|terminal|online), amount, description, reference_id (nullable), created_at
16. **owner_debts** — id, shift_id (FK), amount, is_settled (default false), created_at, settled_at (nullable)
17. **inventory_checks** + **inventory_check_items** — location (shop|warehouse), checked_by, status (in_progress|completed); items: expected_qty, actual_qty, difference

## 5. API Design

Base URL: `/api/v1`. All endpoints require JWT except `POST /auth/login`.

### Response Format

Success:
```json
{
  "data": { ... },
  "meta": { "page": 1, "limit": 20, "total": 150 }
}
```

Error:
```json
{
  "error": {
    "code": "VALIDATION_ERROR",
    "message": "description"
  }
}
```

### HTTP Status Codes

| Code | Meaning |
|------|---------|
| 200 | Success (GET, PUT) |
| 201 | Created (POST) |
| 400 | Validation error |
| 401 | Unauthorized |
| 403 | Forbidden (role) |
| 404 | Not found |
| 500 | Internal error |

### Endpoints

#### Auth
- `POST /auth/login` — login + password → access + refresh tokens
- `POST /auth/refresh` — refresh token → new access token
- `GET /auth/me` — current user info

#### Users (owner only)
- `GET /users` — list
- `POST /users` — create
- `PUT /users/:id` — update
- `DELETE /users/:id` — delete

#### Products (employee: read + create; owner: full)
- `GET /products` — list (?search, ?supplier_id, ?brand, ?page, ?limit)
- `GET /products/:id` — details (includes computed margin)
- `POST /products` — create
- `PUT /products/:id` — update (owner)
- `DELETE /products/:id` — delete (owner)

#### Suppliers (owner only)
- `GET /suppliers` — list
- `GET /suppliers/:id` — details + settlement info
- `POST /suppliers` — create
- `PUT /suppliers/:id` — update
- `DELETE /suppliers/:id` — delete

#### Purchase Orders (owner only)
- `GET /purchase-orders` — list (?supplier_id, ?status)
- `POST /purchase-orders` — create with items
- `POST /supplier-payments` — pay supplier (money / product return)

#### Creditors (owner only)
- `GET /creditors` — list
- `GET /creditors/:id` — details + transaction history
- `POST /creditors` — create
- `POST /creditor-transactions` — receive / repay

#### Clients (employee + owner)
- `GET /clients` — list
- `GET /clients/:id` — details + purchases + debts
- `POST /clients` — create
- `POST /client-payments` — debt payment

#### Shifts + Sales (employee + owner)
- `POST /shifts/open` — open shift
- `POST /shifts/close` — close shift (aggregation + safe + owner debt)
- `GET /shifts/current` — current shift
- `GET /shifts` — history
- `POST /sales` — create sale (split payment)
- `POST /sale-returns` — return product

#### Stock (owner only)
- `POST /stock/transfer` — move product (warehouse ↔ shop)
- `GET /stock/shop` — shop inventory
- `GET /stock/warehouse` — warehouse inventory
- `POST /inventory-checks` — start check
- `PUT /inventory-checks/:id` — update / complete (auto-correct on completion)

#### Safe (owner only)
- `GET /safe/balance` — balances: cash / terminal / online
- `GET /safe/transactions` — movement history
- `POST /safe/owner-deposit` — deposit cash (settle owner debt)

#### Dashboard (owner only)
- `GET /dashboard/revenue?from&to` — revenue for period
- `GET /dashboard/profit?from&to` — profit for period
- `GET /dashboard/stock-value` — total stock value
- `GET /dashboard/sales-by-supplier?from&to` — sales by supplier
- `GET /dashboard/paid-vs-debt` — paid vs debt to clients

## 6. Key Business Logic

### Shift Close
1. Aggregate payment amounts for the shift from sales
2. Write total_cash, total_terminal, total_online, total_debt_sales to shifts
3. Create safe_transactions: cash → cash, terminal → terminal
4. Online: transfer to safe as cash + create owner_debts record
5. Set status = closed

### Product Sale
1. Validate: shift is open, qty_shop >= quantity
2. Validate: paid_cash + paid_terminal + paid_online + paid_debt = total_amount
3. Deduct qty_shop
4. If paid_debt > 0: client_id required, increment client.total_debt
5. Create sales record

### Client Return
1. **Full**: return product to qty_shop, deduct from safe
2. **Exchange**: return old to qty_shop, deduct new from qty_shop
3. **Exchange with diff**: same + money movement in safe
4. If original sale was on debt: reduce client.total_debt

### Supplier Purchase
1. Create purchase_order with items
2. Stock arrival: qty_shop or qty_warehouse per item destination
3. If cash: deduct from safe, status = paid
4. If debt: increment supplier.total_debt, status = unpaid

### Supplier Payment
1. **Money**: deduct from safe, reduce supplier.total_debt, update purchase_order.paid_amount
2. **Product return**: deduct product from stock, reduce debt by cost_price * qty
3. Recalculate purchase_order.status (paid | partial | unpaid)

### Creditor Transactions
1. **Receive**: record currency + rate, add to safe (cash), increment creditor.total_debt
2. **Repay**: record rate, deduct from safe, reduce creditor.total_debt

### Inventory Check Completion
- When status changes to completed: auto-correct qty_shop or qty_warehouse to match actual_qty for each item

## 7. Patterns & Conventions

- **DI**: Constructor injection wired in main.go — `NewXxxService(repo)`, `NewXxxHandler(service)`
- **Transactions**: `pgx.Tx` passed through service layer for multi-table operations (sale, shift close, purchase)
- **Errors**: Custom types (`ErrNotFound`, `ErrValidation`, `ErrForbidden`), handler maps to HTTP codes
- **Money**: `shopspring/decimal` in Go, `DECIMAL(15,2)` in PostgreSQL
- **Margin**: Computed in service layer (`sell_price - cost_price`), not stored
- **Logging**: `slog` with structured JSON, request middleware logs method/path/status/duration
- **RBAC**: `RequireRole("owner")` middleware on protected routes

## 8. Docker & Deployment

- **docker-compose.yml**: api (port 8082) + db (PostgreSQL 16, port 5434)
- **Dockerfile**: Multi-stage — `golang:1.22-alpine` builder, `alpine` runtime
- **Network**: `familycotton-net`
- **Migrations**: goose, run at app startup or via separate command
- **Env vars**: DB_HOST, DB_PORT, DB_USER, DB_PASSWORD, DB_NAME, JWT_SECRET, JWT_ACCESS_TTL (1h), JWT_REFRESH_TTL (720h), SERVER_PORT (8082)
