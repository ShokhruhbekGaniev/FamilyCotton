# FamilyCotton Backend — Design Spec

**Date:** 2026-03-18
**Stack:** Go 1.22+ | chi v5 | PostgreSQL 16 (pgx v5) | goose | JWT | Docker Compose
**Architecture:** Classic Layered (handler → service → repository)

---

## 1. Decisions Log

| Question | Decision |
|----------|----------|
| Implementation approach | 5 phases, sequential |
| total_debt fields | Denormalized, updated on each transaction |
| Employee permissions | Shifts, sales, returns, products (read+create), clients, suppliers (read only) |
| Owner permissions | Full access |
| photo_url | Plain string, no upload |
| JWT tokens | Access: 1h, Refresh: 30 days, stored in DB |
| Pagination | Offset-based: `?page=1&limit=20` (acceptable for expected data volumes) |
| Inventory check completion | Auto-correct stock quantities |
| Exchange rate | User inputs manually per transaction |
| Architecture | Classic Layered (handler → service → repository) |
| Sales model | One sale (receipt) = multiple items via sale_items table |
| Deletion strategy | Soft-delete (is_deleted flag) for entities with FK references |
| Concurrency | CHECK constraints + row-level locking (SELECT FOR UPDATE) on stock operations |

## 3. Implementation Phases

### Phase 1 — Foundation

- Project scaffolding, go.mod, directory structure
- Configuration loading from env
- Docker Compose: Go app (port 8082) + PostgreSQL 16 (port 5434)
- Dockerfile: multi-stage build (golang:1.22-alpine → alpine)
- Goose migrations for all 18 tables (including sale_items, refresh_tokens)
- JWT authentication (login, refresh, logout, me)
- Refresh tokens stored in DB with revocation support
- Users CRUD (owner only, includes password change)
- Middleware: auth (JWT validation), CORS, request logging (slog), RBAC (RequireRole)
- Chi router with base route structure
- Unified error handling and response format

### Phase 2 — Catalogs

- Products CRUD with filters (?search, ?supplier_id, ?brand)
- Suppliers CRUD (employees: read-only access for product creation)
- Clients CRUD (employee + owner)
- Creditors CRUD (owner only)
- Offset-based pagination on all list endpoints

### Phase 3 — Sales Operations

- Shifts: open/close with full business logic (aggregation, safe transactions, owner debts)
- Sales with sale_items: split payments, qty_shop deduction per item, client.total_debt update
- Sale returns: full / exchange / exchange_diff with stock and safe adjustments
- Client payments: debt repayment

### Phase 4 — Procurement & Stock

- Purchase orders with items (stock arrival to shop/warehouse)
- Supplier payments (money / product return)
- Creditor transactions (receive/repay with exchange rate)
- Stock transfers (warehouse ↔ shop)
- Inventory checks with auto-correction on completion

### Phase 5 — Safe & Dashboard

- Safe transactions log (all monetary movements)
- Safe balance endpoint (cash / terminal / online)
- Owner debts + owner deposit
- Dashboard: revenue, profit, stock-value, sales-by-supplier, paid-vs-debt

## 4. Database

18 tables, all using:
- UUID primary keys (generated by PostgreSQL via `gen_random_uuid()`)
- TIMESTAMPTZ for dates (with `DEFAULT NOW()`)
- DECIMAL(15,2) for monetary amounts (except exchange_rate: DECIMAL(15,4))
- `updated_at` on mutable entities (auto-updated via trigger or application)
- `created_by` (FK → users) on transactional tables for audit trail
- `is_deleted` (BOOLEAN DEFAULT false) on entities referenced by FKs
- CHECK constraints: `qty_shop >= 0`, `qty_warehouse >= 0`

### Tables

1. **users** — id, name, login (unique), password_hash (bcrypt), role (owner|employee), is_deleted, created_at, updated_at

2. **refresh_tokens** — id, user_id (FK), token_hash (unique), expires_at, created_at
   > Enables logout and token revocation. Expired tokens cleaned up periodically.

3. **suppliers** — id, name, phone, notes, total_debt, is_deleted, created_at, updated_at

4. **creditors** — id, name, phone, notes, total_debt, is_deleted, created_at, updated_at

5. **clients** — id, name, phone, total_debt, is_deleted, created_at, updated_at

6. **products** — id, sku (unique), name, brand, supplier_id (FK), photo_url, cost_price, sell_price, qty_shop (CHECK >= 0), qty_warehouse (CHECK >= 0), is_deleted, created_at, updated_at

7. **shifts** — id, opened_by (FK → users), closed_by (FK → users, nullable), opened_at, closed_at (nullable), total_cash, total_terminal, total_online, total_debt_sales, status (open|closed)

8. **sales** — id, shift_id (FK), client_id (FK → clients, nullable), total_amount, paid_cash, paid_terminal, paid_online, paid_debt, created_by (FK → users), created_at
   > One sale = one receipt. Payment is at the receipt level.

9. **sale_items** — id, sale_id (FK → sales), product_id (FK → products), quantity, unit_price, subtotal
   > Line items within a sale. qty_shop deducted per item.

10. **sale_returns** — id, sale_id (FK → sales), sale_item_id (FK → sale_items), new_product_id (FK → products, nullable), quantity, return_type (full|exchange|exchange_diff), refund_amount, surcharge_amount, created_by (FK → users), created_at
    > Validation: return quantity <= original sale_item.quantity - already_returned_qty

11. **purchase_orders** — id, supplier_id (FK), total_amount, paid_amount, status (paid|partial|unpaid), created_by (FK → users), created_at, updated_at
    > Removed payment_type column. Payment method determined by actual payments made.

12. **purchase_order_items** — id, purchase_order_id (FK), product_id (FK), quantity, unit_cost, destination (shop|warehouse)

13. **supplier_payments** — id, supplier_id (FK), purchase_order_id (FK, nullable), payment_type (money|product_return), amount, returned_product_id (FK, nullable), returned_qty (nullable), created_by (FK → users), created_at

14. **creditor_transactions** — id, creditor_id (FK), type (receive|repay), currency (UZS|USD), amount, exchange_rate DECIMAL(15,4), amount_uzs, created_by (FK → users), created_at

15. **client_payments** — id, client_id (FK), amount, payment_method (cash|terminal|online), created_by (FK → users), created_at

16. **stock_transfers** — id, product_id (FK), direction (warehouse_to_shop|shop_to_warehouse), quantity, created_by (FK → users), created_at

17. **safe_transactions** — id, type (income|expense|transfer), source (enum — see below), balance_type (cash|terminal|online), amount, description, reference_id (nullable), created_at

**safe_transactions.source allowed values:**
- `shift` — cash/terminal income from closing a shift
- `creditor_receive` — money received from creditor
- `creditor_repay` — money returned to creditor
- `client_payment` — client debt repayment
- `client_refund` — refund to client on return
- `supplier_payment` — payment to supplier
- `purchase_cash` — cash purchase from supplier
- `online_owner_debt` — online payments transferred as cash (creates owner debt)
- `owner_deposit` — owner deposits cash to settle their debt

18. **owner_debts** — id, shift_id (FK), amount, is_settled (default false), created_at, settled_at (nullable)
    > When a shift closes, online payments go to the owner's personal card. The business owes the owner this amount. owner_deposit settles it.

19. **inventory_checks** — id, location (shop|warehouse), checked_by (FK → users), status (in_progress|completed), created_at, completed_at (nullable)

20. **inventory_check_items** — id, inventory_check_id (FK → inventory_checks), product_id (FK → products), expected_qty, actual_qty, difference

### Critical Indexes

- `products.supplier_id`
- `products.sku` (unique, already indexed)
- `sales.shift_id`
- `sales.client_id`
- `sale_items.sale_id`
- `sale_items.product_id`
- `sale_returns.sale_id`
- `sale_returns.sale_item_id`
- `purchase_order_items.purchase_order_id`
- `supplier_payments.supplier_id`
- `supplier_payments.purchase_order_id`
- `creditor_transactions.creditor_id`
- `client_payments.client_id`
- `safe_transactions.created_at`
- `safe_transactions.source`
- `refresh_tokens.token_hash` (unique)
- `refresh_tokens.user_id`

## 5. API Design

Base URL: `/api/v1`. All endpoints require JWT except `POST /auth/login`.

### Response Format

Success:
```json
{
  "data": { ... },
  "meta": { "page": 1, "limit": 20, "total": 150 }
}
```

Error:
```json
{
  "error": {
    "code": "VALIDATION_ERROR",
    "message": "description"
  }
}
```

### HTTP Status Codes

| Code | Meaning |
|------|---------|
| 200 | Success (GET, PUT) |
| 201 | Created (POST) |
| 400 | Validation error |
| 401 | Unauthorized |
| 403 | Forbidden (role) |
| 404 | Not found |
| 500 | Internal error |

### Endpoints

#### Auth
- `POST /auth/login` — login + password → access + refresh tokens
- `POST /auth/refresh` — refresh token → new access + refresh tokens
- `POST /auth/logout` — revoke refresh token
- `GET /auth/me` — current user info

#### Users (owner only)
- `GET /users` — list
- `POST /users` — create
- `PUT /users/:id` — update (including password change)
- `DELETE /users/:id` — soft-delete

#### Products (employee: read + create; owner: full)
- `GET /products` — list (?search, ?supplier_id, ?brand, ?page, ?limit)
- `GET /products/:id` — details (includes computed margin)
- `POST /products` — create
- `PUT /products/:id` — update (owner)
- `DELETE /products/:id` — soft-delete (owner)

#### Suppliers (employee: read only; owner: full CRUD)
- `GET /suppliers` — list
- `GET /suppliers/:id` — details + settlement info
- `POST /suppliers` — create (owner)
- `PUT /suppliers/:id` — update (owner)
- `DELETE /suppliers/:id` — soft-delete (owner)

#### Purchase Orders (owner only)
- `GET /purchase-orders` — list (?supplier_id, ?status)
- `POST /purchase-orders` — create with items
- `POST /supplier-payments` — pay supplier (money / product return)

#### Creditors (owner only)
- `GET /creditors` — list
- `GET /creditors/:id` — details + transaction history
- `POST /creditors` — create
- `PUT /creditors/:id` — update
- `DELETE /creditors/:id` — soft-delete
- `POST /creditor-transactions` — receive / repay

#### Clients (employee + owner)
- `GET /clients` — list
- `GET /clients/:id` — details + purchases + debts
- `POST /clients` — create
- `PUT /clients/:id` — update
- `DELETE /clients/:id` — soft-delete (owner)
- `POST /client-payments` — debt payment

#### Shifts + Sales (employee + owner)
- `POST /shifts/open` — open shift
- `POST /shifts/close` — close shift (aggregation + safe + owner debt)
- `GET /shifts/current` — current shift
- `GET /shifts` — history
- `POST /sales` — create sale with items (split payment)
- `POST /sale-returns` — return product

#### Stock (owner only)
- `POST /stock/transfer` — move product (warehouse ↔ shop)
- `GET /stock/shop` — shop inventory
- `GET /stock/warehouse` — warehouse inventory
- `POST /inventory-checks` — start check
- `PUT /inventory-checks/:id` — update / complete (auto-correct on completion)

#### Safe (owner only)
- `GET /safe/balance` — balances: cash / terminal / online
- `GET /safe/transactions` — movement history
- `POST /safe/owner-deposit` — deposit cash (settle owner debt)

#### Dashboard (owner only)
- `GET /dashboard/revenue?from&to` — revenue for period
- `GET /dashboard/profit?from&to` — profit for period
- `GET /dashboard/stock-value` — total stock value
- `GET /dashboard/sales-by-supplier?from&to` — sales by supplier
- `GET /dashboard/paid-vs-debt` — paid vs debt to clients

## 6. Key Business Logic

### Shift Close
1. Aggregate payment amounts from all sales in the shift (via sale_items)
2. Write total_cash, total_terminal, total_online, total_debt_sales to shifts
3. Create safe_transactions: cash → source `shift`, balance_type `cash`; terminal → source `shift`, balance_type `terminal`
4. Online payments: the owner receives online payments to their personal card on behalf of the business. Create safe_transaction with source `online_owner_debt`, balance_type `cash` (the equivalent cash enters the safe). Create `owner_debts` record for the amount owed to the owner.
5. Set status = closed, closed_by, closed_at

### Product Sale
1. Validate: shift is open
2. For each sale_item: validate qty_shop >= quantity, deduct qty_shop (using SELECT FOR UPDATE to prevent races)
3. Validate: paid_cash + paid_terminal + paid_online + paid_debt = total_amount (sum of all item subtotals)
4. If paid_debt > 0: client_id required (client must be pre-registered), increment client.total_debt
5. Create sales record + sale_items
6. All within a single DB transaction

### Client Return
1. Validate: return quantity <= sale_item.quantity - already_returned_qty for this sale_item
2. **Full**: return product to qty_shop. Refund allocated proportionally to original payment methods (cash from safe cash, terminal from safe terminal, etc.). If original sale was partially on debt, reduce client.total_debt proportionally.
3. **Exchange**: return old product to qty_shop, deduct new product from qty_shop
4. **Exchange with diff**: same as exchange + money movement in safe (refund if new is cheaper, surcharge if new is more expensive)

### Supplier Purchase
1. Create purchase_order with items (all within a transaction)
2. Stock arrival: increment qty_shop or qty_warehouse per item destination
3. If paid_amount = total_amount at creation: deduct from safe, status = paid
4. If paid_amount = 0: increment supplier.total_debt by total_amount, status = unpaid
5. If 0 < paid_amount < total_amount: deduct paid from safe, add remainder to supplier.total_debt, status = partial

### Supplier Payment
1. **Money**: deduct from safe, reduce supplier.total_debt, update purchase_order.paid_amount (if linked)
2. **Product return**: deduct product from stock, reduce debt by cost_price * qty
3. Recalculate purchase_order.status (paid if paid_amount >= total_amount | partial | unpaid)

### Creditor Transactions
1. **Receive**: record currency + exchange_rate (user-provided), compute amount_uzs, add to safe (cash), increment creditor.total_debt (in UZS)
2. **Repay**: record currency + exchange_rate, compute amount_uzs, deduct from safe (cash), reduce creditor.total_debt (in UZS)

### Inventory Check Completion
- When status changes to completed: for each inventory_check_item, set product's qty_shop (or qty_warehouse based on location) to actual_qty. Record difference for audit.

## 7. Patterns & Conventions

- **DI**: Constructor injection wired in main.go — `NewXxxService(repo)`, `NewXxxHandler(service)`
- **Transactions**: `pgx.Tx` passed through service layer for multi-table operations (sale, shift close, purchase)
- **Concurrency**: `SELECT ... FOR UPDATE` on product rows during stock-modifying operations
- **Errors**: Custom types (`ErrNotFound`, `ErrValidation`, `ErrForbidden`), handler maps to HTTP codes
- **Money**: `shopspring/decimal` in Go, `DECIMAL(15,2)` in PostgreSQL
- **Margin**: Computed in service layer (`sell_price - cost_price`), not stored
- **Logging**: `slog` with structured JSON, request middleware logs method/path/status/duration
- **RBAC**: `RequireRole("owner")` middleware on protected routes; `RequireRole("owner", "employee")` for shared routes
- **Soft-delete**: Queries filter `WHERE is_deleted = false` by default
- **Audit**: `created_by` populated from JWT user ID in context
